# post-x スキル

ブログ記事の内容をもとに @totsu_ai_lab 向けのXポストを生成・投稿する。

## 動作モードの判定

引数を確認してモードを決定する：

- `ワークフローモード:` を含む → **ワークフローモード**（generate-postからの呼び出し）
- スラッグ（例: `claude-code-tips`）または URL を受け取った → **スタンドアロンモード**
- 引数なし → ユーザーにスラッグかURLを聞く

---

## ワークフローモード

generate-postから以下の形式で呼び出される：

```
ワークフローモード:
slug: {slug}
title: {記事タイトル}
url: {WordPressのURL}
draft_dir: drafts/{slug}/
```

ヒアリングをスキップして即座に投稿文を生成し、確認を取ってから投稿する。

---

## スタンドアロンモード

### Step 1: 記事内容を取得

**スラッグが渡された場合：**
`drafts/{slug}/` ディレクトリから記事内容を読み込む。

```
優先順位：
1. drafts/{slug}/article.html または drafts/{slug}/article.md
2. drafts/{slug}/ 内の任意の記事ファイル
```

記事のWordPress URLは `drafts/{slug}/` 内のメタデータファイルや投稿済みIDから取得する。
URLフォーマット：`https://www.m-totsu.com/{postId}`（例: `https://www.m-totsu.com/843`）
URLが見つからない場合はユーザーに確認する。

**URLが渡された場合：**
Claude in Chromeで記事ページを開いて本文を読み込む。

## 投稿文の生成

### ストーリー形式

読者が「あるある」と共感できるストーリー構造で書く：

```
[状況・問題] → [試したこと] → [発見・結果]
```

**良い例：**
```
🧪 Claude Codeで議事録を自動化しようとしたら
最初は全然うまくいかなかった

ポイントは「SKILL化」だった
テンプレートとして保存したら
誰でも再現できる形になった

実験の詳細はブログに →
https://www.m-totsu.com/843
```

**悪い例（避ける）：**
- 機能や手順の羅列
- 「〜について解説しました」という告知系
- 専門用語の多用

### サンプル投稿の読み込み

`drafts/x_samples.txt` が存在する場合は必ず読み込み、文体・リズム・語尾・改行パターンを学習してから生成する。
このファイルはユーザーが「これが自分の文体」と決めた参考ツイート集。サンプルがあればそれが最優先の文体指針になる。

### 投稿構成：2ツイート戦略（必須）

Xのアルゴリズムは外部リンクを含むツイートのリーチを抑制する。
そのため **本文とリンクを分けた2ツイート構成** を標準とする：

```
ツイート①（本文）：テキストのみ → アルゴリズム最大活用・リーチ最大化
    └ ツイート②（自分へのリプライ）：URL のみ → OGPカード表示
```

- ツイート①にリンクは含めない
- ツイート②はURLだけでOK（OGPカードが自動表示される）
- ハッシュタグはどちらにも不要

### 生成ルール

1. **絵文字を冒頭に1〜2個**：テーマに合ったもの（🧪⚡💡🤖📝など）
2. **1〜3行で状況・発見を語る**：体験談として書く
3. **改行を活かしてリズムを作る**：1行1メッセージが理想
4. **リンクは本文に含めない**：リプライ用の別ツイートとして分離する
5. **280文字以内**（日本語1文字=1文字でカウント）
6. **ハッシュタグは不要**：コンテンツ勝負で読ませる
7. **3パターン生成**：毎回以下の3タイプを出してユーザーに選ばせる
   - **体験談型**：「〜をやってみたら〜だった」という一人称の流れ
   - **失敗談型**：「最初は〜で失敗した、でも〜で解決した」
   - **驚き型**：「〜が意外にも〜だった」という発見ドリブン

### 文体のトーン

- 話し言葉・体験談ベース（「〜だった」「〜になった」「〜だよ」「〜かも」）
- 上から目線ではなく、一緒に探求する姿勢
- 失敗も正直に書く（「最初はうまくいかなかった」等）
- エンジニアにも非エンジニアにも伝わる言葉
- 「で、」「だから」「そしたら」「やっぱ」「なんか」「ちょっと」など口語の接続詞・副詞を使う
- 体言止めを積極的に使う（「これ大事。」「ほんとそれ。」「マジで良かった。」）
- 疑問形を時々入れる（「これって自分だけ？」「あれ、そういう意味だったの？」）

### Burstiness（文長の揺らぎ）を意図的に作る

AIの文章は文長が均一で「滑らか」になりすぎる。人間らしさは **意図的な揺らぎ** にある：

```
❌ AI的（均一で滑らか）:
Claude Codeを使い始めてから作業効率が大幅に向上しました。
特にSKILL化の機能が便利で、繰り返し作業を自動化できます。
導入も比較的簡単なので、ぜひ試してみてください。

✅ 人間的（短文と長文を混在・揺らぎあり）:
Claude Code、乗り換えてよかった。

試行錯誤のたびにAPI料金が積み上がってた前の環境、
「これ毎月いくらかかるんだ...😭」ってなってたやつ。

月額固定になって、ガシガシ使っても追加料金ゼロ👀
```

**Burstiness実践ルール：**
- 超短い1行（2〜5文字）を意図的に挟む
- その後にやや長めの説明行を続ける
- 全行が似た長さにならないよう意識する
- 1投稿の中で文末の形を変える（「。」「！」「〜だよ」「〜かも」を混在させる）

### 絵文字・記号の使い方（このアカウント固有のスタイル）

**使う絵文字（優先順位順）：**
- `☺️` — 嬉しい・うまくいった・ほっこりした場面
- `😭` — 失敗・驚き・想定外だった場面
- `👀` — 「気になる」「調べてみた」「ちょっと待って」的な場面
- `👉️` — 誘導・「ここがポイント」「続きはこちら」的な場面
- `🧪` — 実験・試してみた系（アカウントテーマに合うとき）

**記号・口語による余韻：**
- `...` または `。。。` — 考え中・余韻・続きがある感じを出すときに使う
- `！` — 発見・驚き・テンションが上がる場面で使う（多用しない）
- `www` — 苦笑い・自虐・「まあそうだよね」的な場面で文末や行末に使う（1投稿に最大1回）
- これらは「間」や「感情の揺れ」を表現するための道具。1投稿に1〜2箇所が自然

**禁止絵文字パターン：**
- 冒頭に🔥💡✅などのビジネス系テンプレ絵文字を並べない
- 行末ごとに絵文字をつけない（AI臭さが出る）

### AI臭さを消す禁止表現

以下の表現は絶対に使わない。これらはAIが好んで使うが人間らしくない：

**禁止ワード（日本語）**
- 〜いただく / 〜いただけます
- 〜することができます / 〜することが可能です
- 〜することが重要です / 〜が求められています
- 〜を活用することで / 〜を活用して
- 一方、〜 / さらに、〜 / また、〜（箇条書き的接続詞）
- 〜推奨します / 〜をお勧めします
- 〜において / 〜に関して / 〜に際して
- 「まとめ」「ポイント」「注目」（SNSで浮く単語）
- 〜でしょうか？（問いかけで終わる締め）
- 非常に〜 / とても〜（副詞の多用）
- 整いすぎた起承転結（完璧な構成はAI臭い）

**禁止パターン**
- 「①②③」などの番号リスト形式
- 「・〜 ・〜 ・〜」の箇条書き形式
- 「本記事では〜を解説しました」という告知文末
- 「ぜひ〜してみてください」という呼びかけ締め
- 全文の長さが均一（Burstinessがない状態）
- 全文末が「。」で終わる単調なリズム

### 生成後の自己チェック（必須）

投稿文を生成したら以下を必ず確認してから提示する：

- [ ] 文の長さにバラつきがあるか（Burstiness）
- [ ] AI定型フレーズが含まれていないか
- [ ] 感情・体験・個人的な気づきが入っているか
- [ ] 文末が「。」だけで単調になっていないか
- [ ] 音読して「友達に話しかけている感じ」があるか

## 確認フロー

生成した投稿文をユーザーに提示し、必ず確認を取る：

```
以下の内容でXに投稿します。よろしいですか？

【ツイート①（本文）】
---
[生成した投稿文]
---
文字数: XX/280文字

【ツイート②（リプライ）】
---
https://www.m-totsu.com/{postId}
---

[投稿する] / [修正する] / [キャンセル]
```

AskUserQuestion を使って確認を取る。
修正依頼があれば投稿文を直して再確認。
OKが出たら投稿へ進む。

## X投稿手順（Claude in Chrome）

### 1. タブを準備

```
tabs_context_mcp() → 現在のタブ一覧を取得
```

X（https://x.com）のタブが既に開いていればそれを使う。
なければ新しいタブを作成して https://x.com/compose/post に遷移する。

### 2. 投稿フォームに入力

```
navigate(url="https://x.com/compose/post", tabId=...) → 投稿画面を開く
```

ページロード後：
```
find(query="tweet text input or post text area") → 入力エリアを特定
computer(action="left_click", ...) → クリックしてフォーカス
computer(action="type", text="{投稿文}") → テキストを入力
```

### 3. 投稿前に文字数を確認

スクリーンショットで入力内容と残り文字数を確認する。
文字数オーバーの場合は投稿せずユーザーに報告する。

### 4. ユーザーに投稿を委ねる

**投稿ボタンはユーザーが押す。** Claude は入力まで行い、以下を報告して完了とする：

```
✅ 投稿文を入力しました。
内容を確認して「投稿する」ボタンを押してください。

文字数: XX/280
```

Claude が投稿ボタンをクリックすることは**しない**。
誤投稿防止のため、最終的な送信は必ずユーザー自身が行う。

---

## 完了報告

```
✅ 投稿文の入力が完了しました
文字数: XX/280文字
あとは「投稿する」ボタンを押すだけです！
```

## エラーハンドリング

- **ログイン未済**: 「@totsu_ai_labでログインが必要です」と報告して停止
- **文字数オーバー**: 自動的に短縮案を提示してユーザーに確認
- **投稿失敗**: スクリーンショットをユーザーに見せて手動対応を依頼
- **URLが取得できない**: ユーザーにブログURLを確認
